## Summary

- `src-tauri/src/commands/agent.rs` に以下 3 つの Tauri commands を実装。
  - `agent_create(state, input: AgentCreateInput) -> AppResult<Agent>`
  - `agent_list_by_project(state, project_id: String) -> AppResult<Vec<Agent>>`
  - `agent_update(state, input: AgentUpdateInput) -> AppResult<Agent>`
- 各 command は薄いアダプタとして `state.with_conn(|conn| db::repo::agent::xxx(conn, ...))` パターンで repo 層に委譲。バリデーション (name 空チェック) / NotFound (project_id 未存在, agent id 未存在) のエラーは repo 層の既存ロジックをそのまま透過。
- 引数名は project commands と揃えた (`state` / `input` / `project_id`)。`agent_list_by_project` のみ引数 1 個のフラット渡し形式 (`{ projectId }`) を想定し、Rust 側は `project_id: String` とした。
- 念のため 3 関数すべてに `#[tauri::command(rename_all = "camelCase")]` を付与し、フロントエンドの camelCase 呼び出しを Tauri v2 の環境/バージョン差によらず保証した (ticket Implementation Notes で検討事項として挙げられていた対応)。これにより `{ projectId }` / `{ input }` 引数キーも camelCase 側で統一される。
- `agent_update` は repo 層が既に `active_worktree_id` を含む全可変フィールドのマージ更新に対応済みのため、command 層では追加ロジック不要。
- Builder への `invoke_handler` 登録は task-1C05 に委ねるため本チケットでは触らない。

## 実装メモ

- `agent_list_by_project` の Rust 側引数名は `project_id` のまま。`rename_all = "camelCase"` を付けたことで Tauri 側の引数キーは `projectId` になり、フロント側は `invoke("agent_list_by_project", { projectId })` で呼べる。
- `with_conn` のクロージャが `|conn| db::repo::agent::list_by_project(conn, &project_id)` と `&project_id` を外部からキャプチャする形になるが、`project_id: String` がクロージャ呼び出し時点で生存しており問題なし。
- project commands に比べ `rename_all = "camelCase"` を明示した差分があるが、`AgentCreateInput` / `AgentUpdateInput` 自体も `#[serde(rename_all = "camelCase")]` を既に持っているため、入力 JSON は完全に camelCase で一貫する。
- repo 層の `find_by_id` (private) は command 層からは使わず、`db::repo::agent::update` が内部で呼ぶ形を維持。command 層はアダプタ責務に徹する。
