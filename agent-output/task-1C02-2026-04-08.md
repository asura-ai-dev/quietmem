## Summary

- `src-tauri/src/commands/project.rs` に以下 3 つの Tauri commands を実装。
  - `project_create(state, input: ProjectCreateInput) -> AppResult<Project>`
  - `project_list(state) -> AppResult<Vec<Project>>`
  - `project_update(state, input: ProjectUpdateInput) -> AppResult<Project>`
- 各 command は薄いアダプタとして `state.with_conn(|conn| db::repo::project::xxx(conn, input))` パターンで repo 層に委譲。バリデーション / UNIQUE 違反 / NotFound のエラーは repo 層の既存ロジックをそのまま透過。
- 引数名は `state` / `input` を採用 (`agent-docs/tauri-commands.md` §Rust コマンド実装パターン / TS 側 invoke 側の `{ input }` と一致)。
- `AppError` enum から `#[allow(dead_code)]` を撤去。`AppError::NotFound` / `AppError::InvalidInput` は repo 層で構築されて本チケットの commands 経由で露出するようになった。
- `AppState::with_conn` から `#[allow(dead_code)]` を撤去。本チケットで初めて commands 層から呼ばれるようになったため。
- Builder への `invoke_handler` 登録は task-1C05 に委ねるため本チケットでは触らない。

## 実装メモ

- `state.with_conn(|conn| db::repo::project::list(conn))` のクロージャ本体は 1 行だが、Rust の型推論では `|conn| db::repo::project::list(conn)` を `fn(&Connection) -> AppResult<Vec<Project>>` へ素直に解決する (`list` の引数が `&Connection` のため、`conn: &Connection` と推論される)。Turbofish 不要。
- Tauri v2 の `#[tauri::command]` は snake_case 関数名をそのままコマンド名として受理するため、`agent-docs/tauri-commands.md` §命名規則の `project_create` 等がそのまま Rust 関数名 = command 名になる。
- `State<'_, AppState>` の lifetime は `tauri::State<'a, T>` の慣習に従い `'_` にした (他 command との一貫性のため)。
- `commands` モジュールの親 (`lib.rs`) では `mod app_state;` が private のままだが、`commands::project` は `crate::app_state::AppState` の `AppState` 型 (pub) に visibility 上到達できる (private module でも crate 内の子孫からは path でアクセス可能)。
- `with_conn` の 2 箇所の `#[allow(dead_code)]` を撤去した結果、cargo check は warning 0 を維持できた。 `Internal` バリアントは `with_conn` 内のロック失敗経路から呼ばれるため dead ではない。`Io` は paths 層、`Db` は repo 層、`NotFound` / `InvalidInput` は repo 層で実用されており、すべて警告なしに通る。
